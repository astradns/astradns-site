# ADR-009: Perfiles de Topología del Agent

- Estado: Aceptado
- Fecha: 2026-03-12

## Contexto

El data path original de AstraDNS asume intercepción node-local (`DaemonSet` + dirección link-local). Ese modelo es excelente para latencia y aislamiento de fallas, pero se vuelve caro en clústeres muy grandes porque cada nodo carga un stack completo de Agent + Engine.

Al mismo tiempo, equipos de plataforma pidieron un perfil de costo más simple, donde la capacidad DNS se dimensione por cantidad de réplicas y no por cantidad de nodos.

## Decisión

Soportar exactamente dos perfiles de topología:

1. `node-local` (predeterminado)
   - Agent como `DaemonSet`
   - semántica link-local/hostPort
   - aislamiento de caché por nodo
2. `central`
   - Agent como `Deployment`
   - `Service` DNS (UDP/TCP 53)
   - escalado por cantidad de réplicas

`dns-pool` se descarta como perfil de primer nivel. El mismo comportamiento de placement se puede lograr en modo `central` con `nodeAffinity` y `topologySpreadConstraints`.

## Racional

- Dos perfiles cubren el espacio práctico de diseño sin explotar la complejidad del chart.
- `node-local` preserva semánticas tipo NodeLocal DNSCache para workloads de baja latencia.
- `central` reduce de forma significativa la huella de recursos a escala.
- Los guardrails operativos se pueden forzar en tiempo de render del template.

## Guardrails y restricciones de seguridad

- `profile=central` + `network.mode=linkLocal` -> Helm `fail` en render.
- `profile=node-local` + patch de CoreDNS + networking no-linkLocal -> Helm `fail` en render.
- En `central`, el PDB usa `minAvailable: 1`.
- En `central`, readiness probes son obligatorias para evitar routing a pods fríos/no listos.
- En `central`, se aplica spread por hostname por defecto para resiliencia anti-co-location.

## Estrategia de target forward en CoreDNS

En modo `central`, el patch de CoreDNS resuelve el target con este orden:

1. `agent.dnsService.clusterIP` (IP fija, preferida para rollouts deterministas)
2. Descubrimiento en runtime del `clusterIP` del Service por el Job de patch

Esto permite despliegues de producción deterministas manteniendo un fallback de auto-descubrimiento.

## Consecuencias

Positivas:

- Mejor camino de adopción para clústeres grandes.
- Menor huella en steady-state cuando no se requiere placement node-local.
- Modelo de trade-offs explícito y documentado entre latencia, caché y costo.

Trade-offs:

- `central` introduce latencia de Service-hop frente a loopback local.
- El caché cambia de per-node a per-replica.
- La migración requiere actualizar de forma controlada el target forward de CoreDNS.
