# ADR-009: Perfiles de Topología del Agent

- Estado: Aceptado
- Decisión: Soportar dos topologías de despliegue — `node-local` (DaemonSet, predeterminado) y `central` (Deployment + Service) — seleccionable mediante un único valor Helm.
- Por qué: El modelo DaemonSet-por-nodo se vuelve costoso a escala (500 nodos = 500 caches). Dos perfiles cubren todos los patrones reales sin complejidad en el chart.
- Consecuencia: Clusters grandes pueden intercambiar latencia sub-milisegundo por ahorro significativo de recursos cambiando al modo `central`.

## Perfiles

| Aspecto | node-local (predeterminado) | central |
|---------|----------------------------|---------|
| Workload | DaemonSet | Deployment |
| Red | link-local / hostPort | ClusterIP Service |
| Destino CoreDNS | `169.254.20.11` | `astradns-agent-dns.ns.svc.cluster.local` |
| Alcance del caché | por nodo | por réplica (compartido) |
| Latencia | < 1 ms (loopback) | ~1–2 ms (in-cluster) |
| Costo | 1 pod por nodo | cantidad fija de réplicas |

## Values

```yaml
agent:
  topology:
    profile: node-local  # node-local | central

  # específico de central
  deployment:
    replicas: 3
    topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule

  dnsService:
    type: ClusterIP
    port: 53
    sessionAffinity: ClientIP
    sessionAffinityTimeoutSeconds: 1800
```

## Guardrails

- `profile=central` + `network.mode=linkLocal` → Helm `fail` (incompatible)
- `profile=central` con `replicas < 2` → Advertencia en NOTES.txt
- PDB para central: `minAvailable: 1`
- DNS Service expone puerto 53 en UDP y TCP

## Destino de Forward de CoreDNS

En modo `central`, CoreDNS reenvía al FQDN del Service en lugar de la IP link-local. Esto es seguro porque CoreDNS resuelve `.svc.cluster.local` mediante su plugin `kubernetes` (watch de la API en memoria), no mediante reenvío DNS — sin dependencia circular.
