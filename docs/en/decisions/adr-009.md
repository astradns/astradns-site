# ADR-009: Agent Topology Profiles

- Status: Accepted
- Decision: Support two deployment topologies — `node-local` (DaemonSet, default) and `central` (Deployment + Service) — selectable via a single Helm value.
- Why: The DaemonSet-per-node model becomes wasteful at scale (500 nodes = 500 caches). Two profiles cover all real-world patterns without chart complexity.
- Consequence: Large clusters can trade sub-millisecond latency for significant resource savings by switching to `central` mode.

## Profiles

| Aspect | node-local (default) | central |
|--------|---------------------|---------|
| Workload | DaemonSet | Deployment |
| Networking | link-local / hostPort | ClusterIP Service |
| CoreDNS target | `169.254.20.11` | `astradns-agent-dns.ns.svc.cluster.local` |
| Cache scope | per-node | per-replica (shared) |
| Latency | < 1 ms (loopback) | ~1–2 ms (in-cluster) |
| Cost | 1 pod per node | fixed replica count |

## Values

```yaml
agent:
  topology:
    profile: node-local  # node-local | central

  # central-specific
  deployment:
    replicas: 3
    topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule

  dnsService:
    type: ClusterIP
    port: 53
    sessionAffinity: ClientIP
    sessionAffinityTimeoutSeconds: 1800
```

## Guardrails

- `profile=central` + `network.mode=linkLocal` → Helm `fail` (incompatible)
- `profile=central` with `replicas < 2` → Warning in NOTES.txt
- PDB for central: `minAvailable: 1`
- DNS Service exposes port 53 on both UDP and TCP

## CoreDNS Forward Target

For `central` mode, CoreDNS forwards to the Service FQDN instead of the link-local IP. This is safe because CoreDNS resolves `.svc.cluster.local` via its `kubernetes` plugin (in-memory API watch), not via DNS forwarding — no circular dependency.
