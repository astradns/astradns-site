# ADR-009: Agent Topology Profiles

- Status: Accepted
- Date: 2026-03-12

## Context

The original AstraDNS data path assumes node-local interception (`DaemonSet` + link-local address). That model is excellent for latency and fault isolation, but expensive in very large clusters because every node carries a full Agent + Engine stack.

At the same time, platform teams asked for a simpler cost profile where DNS capacity is sized by replica count instead of node count.

## Decision

Support exactly two topology profiles:

1. `node-local` (default)
   - Agent as `DaemonSet`
   - link-local/hostPort semantics
   - per-node cache isolation
2. `central`
   - Agent as `Deployment`
   - DNS `Service` (UDP/TCP 53)
   - replica-based scaling

`dns-pool` is intentionally not a first-class profile. The same placement behavior can be achieved in `central` mode with `nodeAffinity` and `topologySpreadConstraints`.

## Rationale

- Two profiles cover the practical design space without chart explosion.
- `node-local` preserves NodeLocal DNSCache-like semantics for low-latency workloads.
- `central` significantly reduces cluster-wide resource footprint at scale.
- Operational guardrails can be enforced at template render time.

## Guardrails and safety constraints

- `profile=central` + `network.mode=linkLocal` -> Helm render `fail`.
- `profile=node-local` + CoreDNS patching + non-linkLocal networking -> Helm render `fail`.
- In `central`, PDB uses `minAvailable: 1`.
- In `central`, readiness probes are mandatory so Service routing avoids cold/unready pods.
- In `central`, default spread by hostname is applied for anti-co-location resilience.

## CoreDNS forward target strategy

In `central` mode, CoreDNS patching resolves the forward target using this order:

1. `agent.dnsService.clusterIP` (fixed IP, preferred for deterministic rollouts)
2. Runtime discovery of Service `clusterIP` by the patch job

This allows deterministic production setups while keeping an auto-discovery fallback.

## Consequences

Positive:

- Better adoption path for large clusters.
- Lower steady-state footprint in environments where node-local placement is not required.
- Explicit, documented trade-off model across latency, cache semantics, and cost.

Trade-offs:

- `central` introduces Service-hop latency compared to local loopback.
- Cache behavior changes from per-node to per-replica.
- Migration requires controlled CoreDNS forward target updates.
