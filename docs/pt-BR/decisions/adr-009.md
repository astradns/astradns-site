# ADR-009: Perfis de Topologia do Agent

- Status: Aceito
- Data: 2026-03-12

## Contexto

O caminho de dados original do AstraDNS assume interceptação node-local (`DaemonSet` + endereço link-local). Esse modelo é excelente para latência e isolamento de falhas, mas pode ficar caro em clusters muito grandes, porque cada nó carrega um stack completo de Agent + Engine.

Em paralelo, times de plataforma pediram uma opção com custo previsível por réplica, não por número de nós.

## Decisão

Suportar exatamente dois perfis de topologia:

1. `node-local` (padrão)
   - Agent como `DaemonSet`
   - semântica link-local/hostPort
   - cache isolado por nó
2. `central`
   - Agent como `Deployment`
   - `Service` DNS (UDP/TCP 53)
   - escala por número de réplicas

`dns-pool` não será perfil de primeira classe. O mesmo resultado de placement pode ser obtido no modo `central` com `nodeAffinity` e `topologySpreadConstraints`.

## Racional

- Dois perfis cobrem o espaço real de decisão sem inflar a complexidade do chart.
- `node-local` preserva semântica tipo NodeLocal DNSCache para workloads de baixa latência.
- `central` reduz bastante footprint em ambientes de larga escala.
- Guardrails de segurança podem ser aplicados no render do Helm.

## Guardrails e restrições de segurança

- `profile=central` + `network.mode=linkLocal` -> `fail` no render.
- `profile=node-local` + patch de CoreDNS + networking sem linkLocal -> `fail` no render.
- Em `central`, PDB usa `minAvailable: 1`.
- Em `central`, readiness probe é obrigatório para evitar roteamento para pods frios.
- Em `central`, spread por hostname é aplicado por padrão para reduzir co-localização.

## Estratégia de alvo de forward no CoreDNS

No modo `central`, o patch job resolve o alvo nesta ordem:

1. `agent.dnsService.clusterIP` (IP fixo, preferível em produção)
2. descoberta em runtime do `clusterIP` do Service

Assim você ganha previsibilidade quando precisa de alvo estável, mantendo fallback automático quando não quer fixar IP.

## Consequências

Positivas:

- melhor caminho de adoção para clusters grandes;
- menor consumo contínuo quando node-local não é obrigatório;
- modelo explícito de trade-off entre latência, cache e custo.

Trade-offs:

- `central` adiciona hop de Service em relação ao loopback local;
- semântica de cache muda de por-nó para por-réplica;
- migração exige atualização controlada do alvo de forward do CoreDNS.
