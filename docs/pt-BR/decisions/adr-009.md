# ADR-009: Perfis de Topologia do Agent

- Status: Aceito
- Decisão: Suportar duas topologias de deploy — `node-local` (DaemonSet, padrão) e `central` (Deployment + Service) — selecionável por um único valor Helm.
- Por quê: O modelo DaemonSet-por-nó se torna custoso em escala (500 nós = 500 caches). Dois perfis cobrem todos os padrões reais sem complexidade no chart.
- Consequência: Clusters grandes podem trocar latência sub-milissegundo por economia significativa de recursos mudando para modo `central`.

## Perfis

| Aspecto | node-local (padrão) | central |
|---------|---------------------|---------|
| Workload | DaemonSet | Deployment |
| Rede | link-local / hostPort | ClusterIP Service |
| Alvo CoreDNS | `169.254.20.11` | `astradns-agent-dns.ns.svc.cluster.local` |
| Escopo do cache | por nó | por réplica (compartilhado) |
| Latência | < 1 ms (loopback) | ~1–2 ms (in-cluster) |
| Custo | 1 pod por nó | contagem fixa de réplicas |

## Values

```yaml
agent:
  topology:
    profile: node-local  # node-local | central

  # específico do central
  deployment:
    replicas: 3
    topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule

  dnsService:
    type: ClusterIP
    port: 53
    sessionAffinity: ClientIP
    sessionAffinityTimeoutSeconds: 1800
```

## Guardrails

- `profile=central` + `network.mode=linkLocal` → Helm `fail` (incompatível)
- `profile=central` com `replicas < 2` → Aviso em NOTES.txt
- PDB para central: `minAvailable: 1`
- DNS Service expõe porta 53 em UDP e TCP

## Alvo de Forward do CoreDNS

No modo `central`, o CoreDNS encaminha para o FQDN do Service em vez do IP link-local. Isso é seguro porque o CoreDNS resolve `.svc.cluster.local` via seu plugin `kubernetes` (watch da API em memória), não via encaminhamento DNS — sem dependência circular.
